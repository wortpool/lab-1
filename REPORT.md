## Merge conflict

Конфлікт було спеціально створено в `README.md`: гілка `develop` і гілка
`feature/conflict-demo` змінили один і той самий рядок у секції `Pull Request`.
Після команди `git merge feature/conflict-demo` Git зупинив merge, показав
конфлікт у `README.md`, після чого конфлікт було вирішено вручну.

Фінальний merge commit:

```text
cec6887 docs: resolve pull request note conflict
```

## Контрольні питання

### 1. У чому різниця між `git merge` та `git rebase`?

`git merge` об'єднує історії гілок і може створити окремий merge commit.
Історія залишається нелінійною, зате добре видно, коли саме гілки
об'єднувалися. `git rebase` переносить коміти однієї гілки поверх іншої,
роблячи історію лінійнішою, але змінює хеші перенесених комітів.

### 2. Чому не можна зберігати `.env` з паролями у репозиторії?

Тому що `.env` може містити паролі, API-ключі, токени доступу та інші секрети.
Якщо такий файл потрапить у репозиторій, секрети можуть побачити інші люди.
Правильний підхід: додати `.env` у `.gitignore`, а в репозиторії залишити
тільки `.env.example` без справжніх секретів.

### 3. Яку роль відіграє Pull Request у QA?

Pull Request дозволяє перевірити зміни до злиття в основну гілку. Через PR
можна провести Code Review, залишити коментарі, знайти помилки, обговорити
архітектурні рішення та переконатися, що зміни не погіршують якість продукту.
